# Bug 猎杀团队协作协议

本协议用于生产 Bug 和线上故障的快速响应、定位和修复。核心原则：**先止血，后分析，最后修复**。

---

## 角色分工

| 角色 | 职责 |
|---|---|
| Incident Commander（指挥官）| 协调整体响应，做关键决策，对外沟通 |
| Bug Hunter（排查员）| 深入日志和代码，定位根因 |
| Hotfix Developer（修复员）| 根据根因快速编写修复代码 |
| Verifier（验证员）| 验证修复效果，确认问题消除 |

> 2人团队时：一人担任 Commander + Verifier，一人担任 Hunter + Developer

---

## Bug 分级标准

| 级别 | 定义 | 响应时限 | 示例 |
|---|---|---|---|
| P0 灾难 | 核心功能完全不可用，影响所有用户 | 立即响应 | 服务全部宕机、数据库连接断开 |
| P1 严重 | 核心功能部分中断，影响 > 10% 用户 | 30 分钟内 | 支付功能不可用、登录概率性失败 |
| P2 一般 | 非核心功能故障，有降级方案 | 2 小时内 | 报表导出失败、邮件通知延迟 |
| P3 轻微 | 功能异常但不影响主流程 | 下一个迭代 | UI 显示错位、提示语错误 |

---

## 响应流程

```
P0/P1: 告警触发 → 快速止血 → 根因分析 → 热修复 → 验证 → 复盘
P2/P3: 问题登记 → 复现确认 → 根因分析 → 常规修复 → 测试 → 复盘
```

### 阶段 0：快速止血（P0/P1 专用，目标 < 10 分钟）

优先考虑以下止血手段（按速度排序）：
1. **回滚**：恢复到上一个已知良好版本（最快）
2. **降级**：关闭故障功能开关，启用备用方案
3. **扩容**：如果是资源耗尽问题，临时扩容
4. **流量切换**：将流量切到备用节点

### 阶段 1：问题上报（Bug Hunter → All）

```
[INCIDENT:INC-037] P1 严重
时间: 2025-03-01 14:23
现象: 登录接口 POST /api/auth/login 返回 500，错误率 23%
影响: 约 15% 用户无法登录
初步判断: 数据库连接相关（日志见 error_id: ERR-4892）
止血状态: 未止血，继续排查
@All 需要支援
```

### 阶段 2：根因分析报告（Bug Hunter → All）

```
[ROOT CAUSE:INC-037]
根因: Redis 连接池耗尽
  - 原因链: 14:18 部署 v2.3.1 → 连接未正确关闭 → 连接泄漏 → 15分钟后耗尽
  - 触发条件: 高并发下新代码 login.py:156 在异常路径未执行 finally 块
  - 证据:
    - Redis 监控: 14:23 active connections = 500/500
    - 日志: "RedisError: max connections exceeded" 首次出现 14:23:47
    - 代码: login.py:156 缺少 try/finally
止血方案: 重启服务释放连接（已执行，服务恢复 14:31）
修复目标: login.py finally 块确保连接释放
```

### 阶段 3：热修复提交（Hotfix Developer → All）

```
[HOTFIX:INC-037]
修复: login.py:156 添加 try/finally 确保 Redis 连接归还连接池
分支: hotfix/inc-037-redis-connection-leak
测试: 本地复现问题并验证修复 ✓  单测全通过 ✓
@Verifier 请验证线上效果
```

### 阶段 4：验证报告（Verifier → All）

```
[VERIFIED:INC-037]
部署时间: 14:52
验证结果:
  - 错误率: 23% → 0% ✓
  - Redis 连接数: 稳定在 12-45 范围内 ✓
  - 登录成功率: 100% ✓
观察时间: 15分钟无复发
结论: 问题已解决，关闭 INC-037
```

### 阶段 5：事后复盘（Incident Commander 主导，故障解决后 24 小时内）

**Post-Mortem 报告模板**：
```markdown
## 事故复盘：INC-037 Redis 连接泄漏

**影响摘要**
- 时间: 14:23 - 14:31（8 分钟）
- 影响用户: ~15%
- 错误率: 峰值 23%

**时间线**
- 14:18  v2.3.1 部署完成
- 14:23  监控告警触发，错误率 > 20%
- 14:25  开始排查
- 14:29  定位根因（Redis 连接泄漏）
- 14:31  回滚止血，服务恢复
- 14:52  热修复部署

**根因**
login.py 异常路径缺少 finally 块，Redis 连接未归还连接池。
高并发下 8 分钟耗尽 500 个连接上限。

**为什么没被测试发现**
单测未覆盖异常路径的连接释放场景。

**改进措施**
- [ ] 添加 Redis 连接数监控告警（阈值 80%）
- [ ] 代码审查 checklist 增加"资源释放"检查项
- [ ] 单测覆盖异常路径的资源清理
- [ ] 考虑使用上下文管理器强制连接管理
```

## 行为规范

- **止血优先**：未止血前不纠结根因，先把损失控制住
- **实时同步**：每 5 分钟在群里更新进度，哪怕是"还在排查"
- **不甩锅**：复盘聚焦系统和流程改进，不针对个人
- **记录一切**：所有命令、发现、决策都记录在事故 timeline
- **单一决策**：同时只有一个人操作生产环境，避免并发操作混乱
